PROJECT OVERVIEW
======================================================================
SID.AI — Full-stack AI chat application with auth, conversation
history, RAG (document upload), web search, and streaming chat.

Tech Stack:
  Frontend: Next.js 16, React 19, TypeScript, Tailwind CSS v4,
            framer-motion, shadcn/ui
  Backend:  Python 3.12, FastAPI, SQLAlchemy (async), SQLite
  Model:    Fine-tuned Qwen2.5-3B-Instruct → GGUF (Q4_K_M, 1.93 GB)
  RAG:      ChromaDB + all-MiniLM-L6-v2 embeddings
  Auth:     JWT (python-jose) + bcrypt
  Search:   Google Custom Search (optional)

Architecture:
  ┌──────────┐      ┌───────────┐      ┌───────────────┐
  │ Frontend │ ───> │  Backend   │ ───> │  llama-server │
  │ Next.js  │ SSE  │  FastAPI   │ HTTP │  (GGUF model) │
  │ :3000    │ <─── │  :8888     │ <─── │  :8081        │
  └──────────┘      └─────┬─────┘      └───────────────┘
                          │
                ┌─────────┼─────────┐
                ▼         ▼         ▼
            SQLite    ChromaDB   Google CSE
            (Auth &   (RAG       (Web
             Chat     vectors)   Search)
             History)

Environment Variables:
  JWT_SECRET_KEY   — JWT signing secret (auto-generated on Render)
  OPENAI_API_KEY   — Groq/OpenAI API key (required for cloud mode)
  OPENAI_BASE_URL  — https://api.groq.com/openai/v1
  OPENAI_MODEL     — llama-3.1-8b-instant
  DATABASE_URL     — sqlite+aiosqlite:////tmp/sidai.db (Render)
  CORS_ORIGINS     — Comma-separated allowed origins
  NEXT_PUBLIC_API_URL — Backend URL for frontend

NOTES & DECISIONS
======================================================================

Why not use the GGUF model for free?
- Model file is 1.93 GB — exceeds ALL free tier limits
- Render free: 512 MB RAM, limited disk
- Railway free: ~512 MB RAM
- Fly.io free: 256 MB RAM, 1 GB disk
- Vercel: serverless functions, 1 GB ephemeral
- Even Hugging Face Spaces CPU (16 GB) would be too slow for chat

Why Groq instead of other free APIs?
- Already configured in .env.example as default
- Free tier: 30 requests/min, 6000 tokens/min
- OpenAI-compatible — no code changes needed
- Very fast inference (LPU hardware)
- Alternatives: OpenRouter (free tier available), Google Gemini API

Why Vercel for frontend instead of Render?
- Vercel is optimized for Next.js (automatic ISR, edge functions)
- No cold starts (Render free tier sleeps after inactivity)
- Faster global CDN
- Better DX with automatic preview deployments

Why SQLite instead of PostgreSQL?
- App already built with SQLite (aiosqlite)
- Render's /tmp directory is ephemeral but lasts while service runs
- For a demo/ChatGPT-like experience, this is acceptable
- To persist data: upgrade to Render's paid tier with persistent disk
  or switch to PostgreSQL (Render offers free 1 GB PostgreSQL)

RAG (document upload) on free tier:
- ChromaDB requires PyTorch (~2 GB install) — too heavy for Render free
- RAG is gracefully disabled when ChromaDB is unavailable
- Upload endpoint still accepts files but skips indexing
- To enable RAG: upgrade Render plan or add ChromaDB as a separate
  managed service
